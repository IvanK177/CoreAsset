# Спецификации CoreAsset

**Тема практики №149** — Сервис учёта рабочих мест с компьютерами: характеристики, ПО, лицензии, история неисправностей

---

## 1. Назначение системы

CoreAsset — веб-система для автоматизации учёта ИТ-активов предприятия. Устраняет хаос в инвентаризации, предотвращает использование нелицензионного ПО, ведёт прозрачную историю обслуживания оборудования.

**Целевая аудитория:** системные администраторы, ИТ-техники, руководители ИТ-отдела.

---

## 2. Функциональные требования

### Модуль A — Реестр сотрудников

| Приоритет | Требование |
|-----------|-----------|
| Высокий | Создание карточки сотрудника: ФИО, должность, отдел, email, табельный номер |
| Высокий | Привязка сотрудника к рабочему месту (1:1) |
| Средний | Смена статуса (активен / уволен); при увольнении рабочее место → «Вакантно» |
| Средний | Поиск по ФИО и табельному номеру |

### Модуль B — Реестр оборудования

| Приоритет | Требование |
|-----------|-----------|
| Высокий | Регистрация ПК: инвентарный номер, серийный номер, тип, кабинет |
| Высокий | Статус жизненного цикла: Активен / В ремонте / Списан / Вакантен |
| Высокий | Привязка к рабочему месту и сотруднику |
| Средний | Характеристики комплектующих: CPU, GPU, RAM, накопители |
| Средний | Фильтрация по статусу и кабинету |

### Модуль C — Пулы лицензий ПО

| Приоритет | Требование |
|-----------|-----------|
| Высокий | Справочник разрешённого ПО (название, версия, вендор) |
| Высокий | Регистрация лицензий: тип (подписка / бессрочная), лимит активаций, дата окончания |
| Средний | Контроль баланса: при установке ПО счётчик уменьшается; при лимите — блокировка |
| Низкий | Красная индикация для пулов с истекающей подпиской (< 30 дней) |

### Модуль D — Журнал неисправностей

| Приоритет | Требование |
|-----------|-----------|
| Высокий | Регистрация инцидента с привязкой к ПК: тип, описание |
| Высокий | Приоритет тикета: Низкий / Средний / Высокий / Критический |
| Высокий | Статус тикета: Открыт → В работе → Исправлен |
| Высокий | Закрытые тикеты не удаляются — архив сохраняется навсегда |

### Модуль E — Сквозные функции

| Приоритет | Требование |
|-----------|-----------|
| Высокий | Поиск по инвентарному номеру, ФИО, IP, MAC-адресу |
| Высокий | Фильтрация по статусу жизненного цикла и кабинету |
| Высокий | Дашборд: количество мест, % исправных, критические тикеты, истекающие лицензии |
| Высокий | Подтверждение перед удалением (модальное окно) |

## 3. Нефункциональные требования

### Безопасность

| Механизм | Реализация |
|----------|-----------|
| Аутентификация | Supabase Auth (email + password), JWT-токены |
| Роли (RBAC) | Администратор (полный доступ) · Техник (без удаления лицензий) · Руководитель (read-only) |
| Row Level Security | RLS-политики на уровне PostgreSQL |
| Каскад | `ON DELETE CASCADE` — ПК → тикеты; `ON DELETE SET NULL` — сотрудник → рабочее место |
| Защита от случайного удаления | Модальное окно подтверждения на каждое удаление |

### Производительность

- Поиск / фильтрация по индексированным полям: **< 200 мс** при 10 000 устройств
- Desktop-first UI: плотные таблицы, минимум графического шума

### Технологический стек

| Слой | Технология |
|------|-----------|
| Фреймворк | Next.js 15 (App Router) |
| Язык | TypeScript |
| UI | Tailwind CSS + shadcn/ui + lucide-react |
| Формы | React Hook Form + Zod |
| БД | Supabase (PostgreSQL) — `@supabase/supabase-js` + `@supabase/ssr` |
| Даты | date-fns |
| Деплой | Vercel |

## 4. Критерии приёмки

| # | Критерий |
|---|---------|
| 1 | Можно создать, просмотреть, изменить и удалить (с подтверждением) запись о ПК |
| 2 | Привязка ПО к ПК уменьшает счётчик лицензий; при лимите — операция заблокирована |
| 3 | Закрытый тикет не удаляется через UI |
| 4 | Неавторизованный пользователь не получает данные (RLS) |
| 5 | Поиск работает быстрее 200 мс на тестовых данных |
| 6 | Приложение задеплоено на Vercel и доступно публично |
