### 1.1 ❌ Миграции Supabase — отмечено «НЕ сделано», фактически СДЕЛАНО

**Позиция в roadmap:** День 11, колонка «Что НЕ сделано» → `❌ миграции Supabase`  
**Позиция в плане задач:** Задача #2 (🔴 Критическое) → «Миграции Supabase — Создать SQL-миграции для всех таблиц + индексы + RLS-политики»

**Фактическое состояние:** Файл [`20260520_proper_rls_policies_and_missing_constraints.sql`](../supabase/migrations/20260520_proper_rls_policies_and_missing_constraints.sql) содержит:
- RLS-политики для всех 7 таблиц (computers, employees, workplaces, software, license_pools, software_installations, incidents)
- Ролевая модель: admin (full CRUD), tech (CRUD minus certain deletes), readonly (read only)

- Index: `idx_computers_room` на `computers(room)`
- CHECK constraint: `license_pools_used_seats_limit` (`used_seats <= total_seats`)
- Trigger: `trg_software_installations_seats` для авто-обновления `used_seats`

**Причина ошибки:** Roadmap был написан до создания миграции, и статус не был обновлен после её реализации.

### 1.3 ❌ RLS-политики — отмечено как отдельная невыполненная задача, фактически СДЕЛАНО

**Позиция в плане задач:** Задача #6 (🔴 Критическое) → «RLS-политики — Row Level Security для всех таблиц (admin: full, technician: CRUD minus licenses delete, manager: read-only)»

**Фактическое состояние:** Полностью реализовано в миграции. Роли: `admin`, `tech`, `readonly`. Политики покрывают SELECT/INSERT/UPDATE/DELETE для всех 7 таблиц с правильными ограничениями:
- `admin` → полный CRUD на всех таблицах
- `tech` → INSERT/UPDATE/SELECT на всех, DELETE ограничен (не может удалять лицензии, может удалять только non-resolved инциденты)
- `readonly` → только SELECT

**Несоответствие:** В roadmap указана роль `manager` (read-only), а в реализации используется `readonly`. Это не ошибка реализации (функционально эквивалентно), но терминологическое расхождение, которое может вызвать путаницу.

**Причина ошибки:** Roadmap не был обновлен после реализации RLS в миграции.

### 2.3 ⚠️ Несоответствие названий ролей: `manager` vs `readonly`

**Описание:** 
- В roadmap (задача #6): «admin: full, technician: CRUD minus licenses delete, **manager**: read-only»
- В user-guide.md: таблица ролей использует «**Руководитель**»
- В реализации (миграция): `get_user_role()` возвращает `'admin'`, `'tech'` или `'readonly'`

Роль `readonly` в коде функционально эквивалентна `manager/Руководитель` из документации, но названия различаются. Это может вызвать путаницу при назначении ролей через `app_metadata`.

**Влияние:** Рассогласование между документацией и реализацией. Если администратор назначит роль `manager` в JWT, `get_user_role()` вернёт `'readonly'` (fallback), что технически работает, но неявно.

**Причина:** Документация писалась на этапе проектирования с терминологией предметной области, а реализация использует технические названия ролей.

### 2.5 ⚠️ Дублирование ответственности: `assignSoftware` обновляет `used_seats` вручную, но триггер тоже это делает

**Описание:** В [`licenses.ts`](../lib/actions/licenses.ts:78-81) server action `assignSoftware` после INSERT в `software_installations` вручную обновляет `used_seats`:
```ts
await supabase.from("license_pools").update({ used_seats: pool.used_seats + 1 }).eq("id", pool.id);
```

Но в БД существует trigger `trg_software_installations_seats` (миграция, строки 167-210), который автоматически инкрементирует `used_seats` при INSERT в `software_installations`.

Это означает, что при каждом вызове `assignSoftware` `used_seats` будет инкрементирован **дважды**: один раз триггером, один раз server action.

Аналогично, `removeSoftware` (строки 87-115) вручную декрементирует `used_seats`, хотя триггер уже делает это при DELETE.

**Влияние:** Критический баг — `used_seats` будет завышен/занижен на 1 при каждой операции привязки/открепления ПО. Это нарушит лицензионный контроль.

**Причина:** Триггер был добавлен в миграции позже, а server actions не были обновлены для удаления ручного обновления `used_seats`.

### Самая критическая ошибка

**Дублирование обновления `used_seats`** (ошибка 2.5) — server action и триггер одновременно инкрементируют/декрементируют `used_seats`, что приведёт к двойному счёту. Это баг реализации, который вызовет некорректное отображение лимита лицензий и потенциально заблокирует привязку ПО к ПК при исчерпании лицензий (или наоборот — позволит превысить лимит). Требует немедленного исправления: убрать ручное обновление `used_seats` из `assignSoftware` и `removeSoftware`, оставив только триггер.
